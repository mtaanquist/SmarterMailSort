SmarterMailSort settings
LLM endpoint

OpenAI-compatible base (Ollama, OpenWebUI, OpenAI). The /v1/chat/completions path is appended automatically.

Standard OpenAI sampling penalty (-2 to 2). A small positive value discourages the model from getting stuck repeating itself. Keep it low — high values penalise the repeated structural tokens of valid JSON (especially with batching) and can corrupt the output. 0 omits it entirely.

Caps how much the model may generate per request, so a runaway response can't burn the whole timeout. "Automatic" scales the cap with your batch size. Set a specific number only if you see decisions being truncated.

How the model is asked to return JSON. "Auto" tries json_object, then json_schema, then plain text, and remembers what your endpoint accepts — leave it on Auto unless you know your endpoint needs a specific mode.

Retries network errors, timeouts, HTTP 429 (rate limit) and 5xx with exponential backoff, honoring a Retry-After header when present. 0 disables retrying.

Classification

Classify from the subject and sender alone, and only download a message's body when the model can't decide without it. Much faster on large folders, and often more accurate — a body's incidental keywords are a common cause of mis-sorting. Turn off to send every message with its body in one pass.

Keep this at 1 for local models to avoid oversaturating the endpoint.

Classify several emails in a single LLM request. Higher values cut round-trips and prompt overhead — a big speed-up on large folders — but need a model that reliably returns well-formed JSON. Start around 10–20 and lower it if results degrade. 1 = one request per email.

Ollama / CORS note

Thunderbird grants this extension cross-origin access to the endpoint host once you save (it requests permission for the origin you enter). Ollama additionally rejects cross-origin requests unless started with OLLAMA_ORIGINS=* (or your extension origin). OpenWebUI and OpenAI need no extra configuration.
